## What happened and why

**One wrinkle worth recording: vite/vitest type duplication under strict TS.** Vitest 2.1.x bundles its own vite transitively, and with `exactOptionalPropertyTypes: true` (plus plain `strict: true`), tsc sees two incompatible `Plugin` / `UserConfig` types and refuses to compile the config. Errors cascade through the `react()` plugin call in `defineConfig({ plugins: [react()] })`.

Tried a few things before landing:
1. **Separate `vitest.config.ts` + `vite.config.ts`.** Didn't help — the clash is between `@vitejs/plugin-react`'s Plugin type and whichever vite `defineConfig` we call, not between the two configs.
2. **`/// <reference types="vitest/config" />`.** Also didn't resolve — the ambient `test` key augmentation worked, but the underlying Plugin-type mismatch remained.
3. **Drop `exactOptionalPropertyTypes`.** Hid the symptom but the Plugin/Plugin mismatch still surfaced even without it.
4. **npm `overrides: { "vite": "^6.0.3" }`** in `package.json`. This is the real fix — forces a single vite version across the dependency tree, so there's only one Plugin type. Removed `exactOptionalPropertyTypes` anyway since the option adds narrow value for a React app (the "undefined-in-optional-field" class of bugs is rare in idiomatic React).

The overrides path is a known workaround for the vitest-bundles-vite situation and should stay in place until vitest's peer story stabilizes. Worth noting so the next dep bump doesn't delete it.

**File naming: kebab-case per CLAUDE.md.** Renamed from the Vite-template default `App.tsx` to `app.tsx`, etc. Biome has `noDefaultExport: "warn"` but exempted for `main.tsx` and `*.config.ts` where the framework requires defaults. All feature code uses named exports.

**Placeholder page content.** Kept the page intentionally thin — just the brand, a one-line "what is this site", and a small status note. The actual UI lands in issues #6 and #7. The footer carries four disclaimers (not medical advice, user-submitted, provenance caveat, not affiliated) — these are load-bearing for the product and belong in a persistent layout, not a per-page block.

**Supabase client setup.** At `src/lib/supabase.ts`. Module-scope env check throws at load time if `VITE_SUPABASE_URL` or `VITE_SUPABASE_ANON_KEY` is missing — broken deploys fail loudly on first import rather than as confusing "network error on submit" later. `persistSession: false, autoRefreshToken: false` because we're not using Supabase auth (anon-only site).

**Dev server port drift.** Running locally, ports 5173 and 5174 were in use (stray from earlier issue-#3 testing or the user's other Supabase project). Vite auto-stepped to 5175. Not a blocker but worth noting: the `APP_BASE_URL` baked into the Edge Function CORS check is hardcoded to `5173`. If we frequently hit port drift in dev, we'll want either a stable way to kill stale processes or a more permissive dev CORS setting.

**Biome is the single lint+format tool.** No ESLint, no Prettier, no Stylelint. `npm run lint` / `npm run lint:fix` / `npm run typecheck` / `npm test` are the four scripts. `build` does typecheck + vite-build in sequence so CI is a single command.

**CLAUDE.md workflow loop followed as written.** Wrote the component tests, ran typecheck + lint + vitest + build, then manually curled the dev server to confirm the HTML shell serves. No subagent review — this is scaffolding, not a security-sensitive change per the CLAUDE.md carve-out.

## Decisions

- **`npm overrides` forcing vite 6** — cleanest fix for the vitest/vite Plugin-type mismatch. Alternatives (downgrading vite, relaxing TS) were worse.
- **`exactOptionalPropertyTypes` dropped** — narrow value for a React app; the vite/vitest friction it caused outweighed the catch rate.
- **Single `vite.config.ts`** with `test` block over a separate `vitest.config.ts` — fewer files, one entry for `defineConfig`, test settings live next to dev-server settings.
- **Disclaimers in the footer as a persistent layout element** — they apply to every page and need to be visible without scrolling to a specific section. Not a per-page React component.
- **Supabase client throws at module load on missing env** — surfaces config errors on page load, not on first network call.
- **Kebab-case file names** — matches CLAUDE.md; renamed from Vite-template's `App.tsx` pattern.
- **No routing yet** — issues #6 and #7 will add react-router (or similar) when there's more than one page to route between. Avoiding premature abstraction.
