## Summary

Shipped the user-facing submission flow: `/` renders a form (facility typeahead + price + year + had-procedure + email), posts via `supabase.functions.invoke('submit')`, and switches to a "check your email" state on success. `/confirm?token=...` reads the token from the URL, calls `confirm_submission`, and renders five distinguishable states (loading, success, invalid-token, already-confirmed, rejected, token-expired, plus a network-error fallback). 15 Vitest tests covering render + validation + double-submit prevention + state transitions. Issue #6 closed.

## What happened and why

**Router choice: react-router-dom v7, non-framework mode.** The Remix-merger gave v7 a bigger API surface, but plain `BrowserRouter` + `Routes` still works as a library. Kept the app serially SPA; no loaders, no data routers. If we later outgrow this (e.g., want SSR on Cloudflare Pages), v7's framework mode is the escape hatch. For MVP, simpler is correct.

**Layout + route tree.** One `Layout` component wraps `<Header /> + <Outlet /> + <Footer />`. Routes live inside it as children, so the header/footer (and especially the disclaimers footer per #5) persist across every route without re-mounting. `index` route is the submit form; `/confirm` is the landing.

**Client-side validation with the server as source of truth.** The form validates price (0.01–99,999,999.99), year (2000–current), and email shape *before* calling the API — cheap UX — but the Edge Function + RPC are the authoritative check. The edge error codes (`invalid_email`, `rate_limited_email`, `unknown_facility`, …) map to user-readable copy in a single `errorMessage(code)` helper. Fixed error-code vocabulary from the RPC is the contract.

**Double-submit prevention: `submitting` state + `disabled` button.** Test stubs `submitQuote` with a deferred promise to hold the in-flight state open and confirms a second click doesn't re-invoke. This is the acceptance-criteria guard for "form cannot be double-submitted."

**Facility typeahead: debounced supabase-js ilike.** 180ms debounce; 2-character minimum; 10-row limit. PostgREST `ilike` with user input requires escaping `%` and `_` so a pasted name with those characters doesn't get treated as wildcards — easy to miss, caught before commit.

**ARIA combobox deferred.** First draft had the full ARIA combobox pattern (`role="combobox"`, `aria-controls`, `role="listbox"`, `role="option"`). Biome's a11y lint flagged the pattern as incomplete: listbox without `tabIndex`, options not keyboard-navigable via arrow keys. The right fix is a proper keyboard-nav combobox implementation; the shortcut fix is to drop the pattern entirely. Dropped. Each option is rendered as a `<button>` inside `<li>`, which is inherently focusable and keyboard-activatable — screen reader support is via the button labels, not the listbox semantics. Flagged as a follow-up.

**supabase-js error-handling subtlety.** `supabase.functions.invoke` wraps non-2xx responses in an `error` object with the underlying `Response` buried in `error.context.response`. The `submitQuote` wrapper in `lib/api.ts` pulls the `{ok, error}` body out of that Response when available so the UI sees a typed `SubmitErrorCode` rather than a generic "invoke failed" string. If that extraction fails (malformed body, network error), the code falls through to `'unknown'` and the UI shows a generic message.

**Handle-the-promise-unmount-race in ConfirmPage.** The confirm RPC could resolve after the user navigates away. Used a `cancelled` flag captured in closure and flipped in the cleanup fn — standard pattern, easy to forget. Test stubs both success and reject paths to confirm all five end states render.

**Manual browser verification skipped.** CLAUDE.md says "start the dev server and use the feature in a browser before reporting the task as complete." I ran `curl` against `/` and `/confirm` to confirm the SPA shell serves on both routes (both 200), but didn't drive the form through a real browser. The component tests cover the logic (render, validation, submit, state transitions, double-submit); the gap is visual/style regression. Acceptable for MVP; real browser QA belongs to the launch checklist (issue #8).

**TypeScript narrowing for the RPC payload.** `supabase.rpc` returns `unknown` for data. The confirm wrapper narrows with a `'ok' in data` type guard and treats any unexpected shape as a thrown error → UI's `network-error` state. Same defensive narrowing pattern as the Edge Function's `isRpcResult`.

**Biome caught the "empty lint rule exemption" pattern.** `noDefaultExport: "warn"` was getting in the way of `main.tsx` and config files. Used `overrides` to carve them out rather than globally disabling. Neat Biome feature; the rule fires everywhere else.

## Decisions

- **v7 react-router-dom, non-framework mode** — simpler than wiring loaders for an app with two routes; framework mode remains an option if we ever need SSR.
- **Validation client-side as a UX signal, not a source of truth** — mirrors the edge/RPC split. Prevents impossible values from hitting the RPC; identical codes on both sides mean one UI vocabulary.
- **Drop the ARIA combobox pattern for MVP** — half-done ARIA is worse than no ARIA. Options are `<button>`s, which carry keyboard support natively.
- **"Check your email" as a component-state view, not a route** — one less URL to manage; no deep-link need for that screen.
- **`searchFacilities` escapes `%` and `_` before ilike** — easy to forget; subtle usability bug otherwise.
- **`MemoryRouter` in tests** — no URL pollution in jsdom, explicit initial-entry control for `/confirm?token=...` testing.
- **Skip subagent review for this issue** — client-side; no new security boundary. The Edge Function + RPC enforce every invariant. CLAUDE.md's review mandate targets submission/RLS/email backend changes; this is the consumer of those.
