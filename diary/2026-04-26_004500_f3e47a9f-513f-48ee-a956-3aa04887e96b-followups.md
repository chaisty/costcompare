## What happened and why

**Search filters expansion (`4248ad1`).** Added `Source` (rate type) and `Year` dropdowns alongside the existing `State` filter. The RPC already accepted these params from issue #4 — only the UI was missing them. Year picker is a fixed list of `currentYear..currentYear-5` rather than a free-form range; keeps the UI tight and the test surface small. Single-year selection maps to `year_from === year_to`.

**Keyset pagination (`5decd54`).** Real fix for the OFFSET-drift caveat that has been carried forward since #4. Wrinkle: the original `search_rates` deliberately omits `rates.id` from the response because for cash rates `rates.id == submissions.rate_id`; using `id` as the keyset cursor would re-leak it. Solution was to add a separate `rates.search_token` column (96-bit `gen_random_bytes`, totally independent from rates.id and any submitter-derived value). Order by `(price, search_token)`, expose via an opaque base64 cursor that encodes `{p, t}` JSON. Volatile DEFAULT triggers per-row rewrite for existing rows, so each gets a fresh token instead of a single shared default.

Cursor decode: anon-supplied input, hardened against malformed payloads — 200-char outer length cap, nested `BEGIN ... EXCEPTION WHEN OTHERS` blocks for base64/JSON/numeric-cast errors, explicit type/null/length checks. All bad paths return the stable `invalid_cursor` error code. 5 tests cover the bad-cursor variants.

**`_search_cursor_encode` helper false start.** First draft factored cursor encoding into a helper function. anon doesn't have EXECUTE on it (private namespace), but `search_rates` is `security invoker` — when anon called `search_rates`, it hit `permission denied for function _search_cursor_encode` because the inner call ran as anon too. Two fixes available: `SECURITY DEFINER` on the helper, or inline the encode. Inlined it — single call site, simpler grant model, tighter blast radius.

**Subagent review caught one stale handler** — `errorCopy` switch in `search.tsx` still had `case 'offset_too_large'` after the RPC stopped emitting it, with no handler for the new `invalid_cursor`. One-line cleanup landed as `86f3a75`.

**NPPES Type-2 backfill (`c24a9c7`).** Architecture briefly looked like a re-pivot — the project explicitly rejected bulk NPPES during #13 in favor of CTSS + cache-on-submit. Asked the user to confirm scope (three flavors offered: full taxonomy-filtered backfill, NPI-enrichment-of-existing-only, curated short-list). They picked flavor 1.

Filter is intentionally narrow — five NUCC taxonomy codes mapping to procedure-relevant org types (ASC, Pain Medicine Clinic, Multi-Specialty Clinic, Surgical Specialty Hospital, General Acute Care Hospital). Expected volume in US is ~17K rows, easily fits in memory. Filter is by code, not label, so the mapping table is small and stable across NPPES releases.

**No auto-download.** NPPES distributes the monthly file as a multi-GB ZIP from a CMS landing page that doesn't expose a stable per-month URL, and `adm-zip` (already a dep for the medicare-rates pipeline) loads ZIPs entirely into memory. Adding `yauzl`/`unzipper` for streaming ZIP extraction would have been ~30 LOC and a new dep for a one-time-per-month chore. Punted to "manual extract + `NPPES_CSV_PATH` env var" — README has the exact CMS landing page link and `unzip` step.

**Crosswalk semantics.** For each NPPES candidate: NPI already in facilities → skip; same lower(name)+state matches an existing row with no NPI → set the NPI on that row (preserves POS-sourced rows' `external_id` and Medicare-listed badge); same lower(name)+state matches a row with a *different* NPI → record as `ambiguousNameStateMatch` and skip (don't merge silently). The decision logic is in a pure `planNppesUpsert` function so it's unit-testable without a Supabase mock.

**Test caught a real bug.** "Two NPPES candidates name-matching the same existing POS row" — both updates were getting written, last-NPI-wins on the same row. Fix: after a crosswalk update, mutate the local-map view of the matched row to set `npi`, so a second candidate falls into the ambiguous bucket instead of double-writing. Wouldn't have noticed without the test.

**Playwright smoke (`2e0838b`).** chrome MCP is unusable on this app (saved memory entry from earlier). Playwright uses CDP directly and doesn't share `document_idle` waits, so it works. Set up:

- `web/e2e/` for specs, Playwright's own runner
- `vite dev --mode test --port <PORT> --strictPort --host 127.0.0.1` as the webServer command

**Two surprises during Playwright setup:**

1. **Windows IPv4/IPv6 mismatch.** vite's default `localhost` binding resolves to `::1` on Windows; Playwright's HTTP healthcheck targets IPv4 by default. Pinned both to `127.0.0.1` explicitly.
2. **Port 5180 was already in use.** This was the prompt for the user's "scan for a clear port" feedback. Saved a memory entry. Bumped to 51234 (verified free at commit time, in user-ephemeral range > 49152) paired with `--strictPort` so a future collision fails loudly instead of falling through to a different port like vite's default behavior would.

Three orphaned vite dev servers (5173, 5174, 5175) were the actual root cause; killed them via `taskkill /F /PID` after the user's "do it" green light.

## Decisions

- **`rates.search_token` instead of `rates.id` as cursor key:** preserves the defense-in-depth that prevents `rates.id` (== `submissions.rate_id` for cash rows) from becoming a public correlator. New column has zero submitter linkage.
- **Inlined cursor codec:** simpler than `SECURITY DEFINER` helper, single call site doesn't justify the indirection.
- **NPPES manual extract over auto-extract:** quarterly chore, not worth a streaming-ZIP dep just to remove one shell step.
- **NPPES taxonomy filter is narrow on purpose:** five codes covering CPT 64628's likely org types. Expanding scope = expand the codes table; the volume cap is policy, not technical.
- **Single Year dropdown over Year From/To:** the RPC takes a range but cash-pay quotes are recent-skewed; `year_from = year_to = picked` is sufficient.
- **`vite dev --mode test`** for Playwright over `vite preview`: skips a build step, faster iteration, HMR doesn't break Playwright's wait strategies (unlike chrome MCP's `document_idle`).
- **port 51234 + `--strictPort`:** fail loudly on collision, fix at the user level. Did not invest in dynamic port-scanning at config-load time — `--strictPort` plus a clear netstat-the-port instruction in the config comment is enough for a solo dev workflow.
- **Skipped CI integration for Playwright:** intended as the immediate next step in this same session.
