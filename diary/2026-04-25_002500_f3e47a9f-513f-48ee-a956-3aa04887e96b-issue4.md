# Issue #4: search_rates RPC for public rate search

**Session:** `f3e47a9f-513f-48ee-a956-3aa04887e96b` (continuation)
**Model:** Claude Opus 4.7
**Date:** 2026-04-25 (UTC; local 2026-04-24 evening)

## Summary

Shipped `search_rates(procedure_code, state, rate_type, year_from, year_to, limit, offset)` as a `SECURITY INVOKER` Postgres RPC exposed via PostgREST. Takes bounded inputs, returns a jsonb envelope with `has_more` pagination, and enforces a column allowlist so the public search surface can't accidentally leak a submitter-derivable field via a future schema change. Issue #4 closed.

## What happened and why

**RPC over direct PostgREST table access.** PostgREST already auto-exposes `rates` to anon via RLS. The lazy option would be: let the frontend query `/rest/v1/rates?procedure_codes=cs.%7B64628%7D&state=eq.CA...`. Rejected because "bounded inputs validated at the boundary" is an acceptance criterion, and PostgREST's query DSL is unbounded. An RPC makes SQL the single source of truth for bounds + column allowlist + shape. Costs a migration; saves a class of future bugs.

**SECURITY INVOKER, not DEFINER.** Every other RPC in this project is DEFINER (writes need elevation). This one is a read path, and RLS on `rates` + `facilities` already allows anon SELECT, so INVOKER is both simpler and future-proof — a later RLS tightening propagates automatically. DEFINER would bake "rates are public" into the function body and drift from the table policy.

**Explicit column allowlist instead of `select *`.** Three columns omitted on purpose:
- `source_submission_id` — internal FK into private submissions.
- `created_at` — hides activity timing; not useful to callers.
- `rates.id` — this one was caught in review. For cash rates, `rates.id == submissions.rate_id` (see `confirm_submission` body). Today anon can't read submissions, so there's no active correlation attack, but exposing `id` bakes the coupling into the public surface. If rate_id ever leaks via another path, or if we later add a "share this rate" feature, the correlation vector becomes real. Cheap defense now; hard to undo later. Frontend doesn't need it for MVP.

**Column allowlist is a pgTAP assertion, not just a comment.** The issue's acceptance criterion said "automated test greps every response shape for `email`" — the initial tests did exactly that (regex + "the word email" + the seeded address). Review pointed out this is regex-brittle: a future `notes` column carrying email content would not be caught. Added a `jsonb_object_keys` equality assertion against an explicit array of 13 expected keys. Any new column that ends up in the response breaks this test by name.

**`has_more` via fetch-one-extra.** Initial cut returned only `limit` and `offset`, leaving the frontend to probe. Review flagged. Fixed by `limit + 1` in the subquery; if the resulting array exceeds `v_limit`, drop the last element and set `has_more=true`. Cheaper than `count(*)`, which would scan the full filtered set.

**Pagination-under-concurrent-insert is still unstable.** `ORDER BY (price, id)` is deterministic within a snapshot but a new lower-priced row inserted between page-1 and page-2 shifts the offset. Documented in a code comment rather than fixed — the fix is keyset pagination (`where (price, id) > (last_price, last_id)` and no offset), which is a different request shape and a larger change. T-in-C ingest may force this later.

**PostgREST returns HTTP 200 for ok-false.** Consistent with the `submit` endpoint: the transport succeeded, the application-level result is in the body. The frontend has to branch on `body.ok`, not the HTTP status. Worth flagging so issue #7's UI code knows not to rely on status codes for user-visible errors.

**E2E:** manual curl against `/rest/v1/rpc/search_rates` with the local publishable key confirmed `has_more`, error paths (`invalid_state`, `invalid_year_range`, `offset_too_large`), and the absence of omitted columns. Re-seeded a single medicare row via `docker exec psql` after `supabase db reset` wiped the stack; didn't bother re-running the full CMS ETL (not needed to validate the function).

**Subagent review** was high-value again. Caught the `rates.id` correlation concern (the single most consequential finding — prompted a defense-in-depth change, not just a doc note), the brittle no-email-leak test (got the column allowlist assertion), and the missing `has_more`. Also caught the `search_path` inconsistency (`public` vs `public, extensions`) and the REVOKE pattern drift from `from public` to `from public, anon, authenticated`. All fixed before commit.

## Decisions

- **RPC over raw PostgREST table access** — bounds live in SQL, single source of truth for the public shape. Trades a migration for durable invariants.
- **SECURITY INVOKER** — the function doesn't need elevated privileges; letting RLS apply naturally is both simpler and future-proof.
- **Omit `rates.id` from the response** — defense in depth against a future cash-rate → submitter correlation via `submissions.rate_id`, even though no active attack exists today. If a stable public handle is ever needed, add an opaque separate id.
- **Column allowlist assertion, not just a grep** — enforces shape invariantly, catches "accidentally added a column" in the failure output by name.
- **`has_more` via `limit + 1`** over a separate `count(*)` — O(limit) instead of O(matches), and the frontend can chain pages without a probe.
- **`ORDER BY (price, id) + offset` accepted** for MVP despite instability under concurrent insert. Keyset pagination is the right endgame; not worth the API change until ingest volume shows it.
- **Reject offset > 10000** to prevent deep-scan DoS. A realistic user won't paginate that far in a single-procedure search.
