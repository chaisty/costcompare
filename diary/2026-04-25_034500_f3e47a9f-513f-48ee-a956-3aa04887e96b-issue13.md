# Issue #13: provider-or-facility submissions, CTSS-backed pickers

**Session:** `f3e47a9f-513f-48ee-a956-3aa04887e96b` (continuation)
**Model:** Claude Opus 4.7
**Date:** 2026-04-25 (UTC)

## Summary

Patients now identify their procedure by **provider, facility, or both**. Schema makes both nullable with a CHECK that at least one is set. Submission frontend uses NLM Clinical Table Search Service (CTSS) for autocomplete on both axes — no bulk NPPES download. Edge Function upserts the selected NPI into local `providers`/`facilities` cache tables on submit and resolves to UUID before calling `submit_quote`. Search rows render provider info when present. Issue #13 closed across four commits: 0519c55 (phase A), bde09d5 (phase B-1), 212e628 (phase B-2), cebc1e7 (phase C).

## What happened and why

**Original plan was wrong-shaped.** First sketch was a bulk NPPES Type-1 + Type-2 ETL (~8GB monthly file, streaming parse, dedup against POS-sourced ASCs). User shared research from another agent suggesting NLM's Clinical Table Search Service — an autocomplete-friendly hosted lookup over the same NPI registry. Pivoted to CTSS + cache-on-submit. Saved a multi-day ETL build, removed a monthly refresh chore, removed ~30GB of disk usage, and got fresher data.

**Phase A landed before the pivot but is reusable.** Phase A's `providers` table was originally going to be ETL-populated; under the new design it's a write-through cache for selections made via CTSS. No schema changes were thrown away — just the data-population path changed. The `submissions.facility_id → nullable` + CHECK + `submissions.provider_id` + RPC rewrite all still apply.

**Trust boundary on display fields.** Frontend gets display data from CTSS (NPI registry is public) and forwards it to the Edge Function on submit. Edge Function trusts the frontend's display only on FIRST insert for a given NPI — subsequent submissions for the same NPI keep what's already there. A bad actor can prime the cache with a fake name on first sight, but every subsequent submitter for the same real NPI sees that poisoning frozen — they can't keep injecting fakes. We could re-validate against CTSS server-side on every insert, but that's a hot-path roundtrip that doesn't change much: NPI → name binding is already public.

**POS-CTSS crosswalk is a quiet feature.** When a CTSS-sourced facility upsert finds an existing POS-sourced row by name+state (which has `external_id` set, `npi` null), the upsert UPDATES the existing row to fill in `npi` rather than inserting a duplicate. A facility that has both `external_id` and `npi` set is a "Medicare-certified ASC cross-validated against NPPES" — a stronger signal than either source alone. The frontend doesn't surface this badge yet; logged for Phase 2.

**`facility_type` from CTSS is lossy.** CTSS's "provider_type" column is a taxonomy-derived label like "Clinic or Group Practice" or "Ambulatory Surgical." Mapping this to our `facility_type` enum (`asc | hospital | medical_center | clinic | other`) is one of those things that's 80% trivial regex and 20% nuance. For MVP, all CTSS-sourced facilities default to `'other'`. POS-sourced rows keep their `'asc'` tag. Worth a follow-up to do a proper taxonomy → enum mapping.

**Search filter on state matches either side.** `search_rates`'s `p_state` filter now matches `f.state = v_state OR p.practice_state = v_state`. Provider-only rates show up under the submitter's state. Rows with neither set (national Medicare) are excluded from state-filtered queries but appear in "all states" — same as before. Tested explicitly in pgTAP.

**Tests follow the change.** pgTAP test count went 70 → 90 (+20 in `provider_submissions.test.sql`). Deno test count went 40 → 54 (+14 across NPI-shape parsing). Vitest count went 22 → 24 (+2 for provider rendering and the at-least-one validation, balanced by simplification of removed tests). Column-allowlist test in `search_rates.test.sql` updated to include the five new `provider_*` keys; the assertion catches any unintended addition.

**One small Vitest thing.** When the frontend sends `{ ...(facility ? { facility: {...} } : {}) }`, the spread-with-empty-object pattern keeps the type clean (no `undefined` properties on the SubmitRequest). Tested both provider-only and provider+facility paths to lock the shape.

**Browser smoke test still skipped.** Component tests cover the picker behavior (debounce, abort, selection, results rendering); E2E via curl confirms the end-to-end NPI upsert path works against the running stack. A real browser session driving the form through CTSS + email confirmation + search is on the launch checklist (issue #8).

## Decisions

- **CTSS over bulk NPPES** — saves a multi-day ETL build and ~30GB disk; freshness is the API's problem, not ours. Falls back to bulk download only if CTSS reliability proves an issue.
- **Cache-on-submit, not on-search** — the `providers` and `facilities` tables get populated as users actually submit. Stale rows for never-confirmed submissions self-clean via the existing token-expiry path.
- **First-write-wins on cache display fields** — first submitter with a given NPI seeds the display; subsequent submissions don't update. Mitigates display-field poisoning at the cost of accepting whichever name the first submitter typed.
- **POS+CTSS crosswalk by name+state** — preserves the "Medicare-certified ASC" signal from POS without forcing CTSS-sourced rows to also resolve a CMS provider number.
- **`facility_type='other'` default for CTSS-sourced rows** — taxonomy-to-enum mapping is a fiddly nuance that doesn't block UX. Filed as follow-up.
- **Frontend sends NPI + display, Edge Function does the upsert** (rather than the frontend hitting Supabase directly to insert). RLS-on-facilities is `SELECT to anon`, not insert; the service-role client in the Edge Function is the only path that can insert. Single source of write authority.
