## What happened and why

**Scope held.** The pre-compact `PLAN.md` anticipated most of the work. Three Deno files (`index.ts`, `validation.ts`, `email.ts`), two test files, one `.env.example`, all committed as one change. No scope creep; the `/confirm?token=...` landing page is still issue #6's problem.

**Edge vs. RPC validation split — worth being explicit.** The RPC already has belt-and-suspenders validation for every field (email shape, price > 0, year range, facility existence, rate limits, advisory lock). Duplicating that at the edge adds drift risk (two sources of truth for email regex / year ceiling) without adding safety. Decision: edge does **structural** checks only (is it the right JS type, does it parse, is the UUID a valid shape); RPC owns **semantic** checks. The edge error codes that overlap the RPC's vocabulary (`invalid_email`, `invalid_procedure_codes`, etc.) are intentionally identical so clients see one code space. One genuine edge-only code: `invalid_body` for non-JSON / non-object payloads.

**Numeric bounds at the edge, after review.** Initial cut relied on the RPC for `invalid_price` / `invalid_year`. Review pointed out that `quoted_price > 99999999.99` would hit the `numeric(10,2)` column and come back as a Postgres overflow → `internal_error`, not `invalid_price`. Edge now caps at `0.01 <= price <= 99999999.99` (column-matched) and `2000 <= year <= 2100` (looser than RPC's `<= current UTC year`, since RPC has `now()` and edge shouldn't). Keeps CLAUDE.md's "bounds at the API boundary" invariant honest.

**EMAIL_MODE=dev-console over "unset key ⇒ console fallback".** First draft had `selectEmailSender` fall back to `ConsoleSender` whenever `RESEND_API_KEY` was unset. Realized that if a prod deploy ever lost that env var, the function would silently start logging confirmation URLs (containing plaintext tokens) to stdout. Added an explicit `EMAIL_MODE=dev-console` opt-in. Unset in prod ⇒ function throws at sender-init ⇒ 500 to client ⇒ pages us. Unset in dev ⇒ same error unless `.env` has the opt-in. This is a small but meaningful hardening over the PLAN.md "Option A" sketch.

**Email-send-failure-after-RPC-success.** The RPC already claimed the per-email/per-IP rate-limit slot before we try to send. If Resend times out at that point, rolling back would need a second RPC call (we don't have one) and the user can't retry (rate-limited). Chose: log the failure server-side, still return `{ok: true}` to the client. Token ages out in 48h if never used; a future resend-confirmation endpoint can recover. This is documented in a code comment because the silent-success on failure is non-obvious.

**RPC result narrowing — subtle bug caught by review.** Initial code had `const result = data as RpcResult;` — a blind cast from `unknown`. If a future RPC change ever returned `null` or a missing-field shape, `result.ok` would be `undefined`, `!result.ok` truthy, and the client would see `{"ok":false}` with no error code (since `data.error` is also `undefined`). Added an `isRpcResult` type guard; unexpected shape now returns `internal_error` instead of a malformed error-without-code.

**IP spoofing in non-Supabase deployments.** On hosted Supabase the platform proxy rewrites `X-Forwarded-For` so first-hop is authoritative. If we ever self-host the Edge runtime directly exposed, `X-Forwarded-For` becomes client-supplied and the per-IP rate limit is trivially bypassable (per-email limit still applies). Added a comment on `firstForwardedIp` documenting this — if we self-host, we must front with a trusted proxy that strips and resets XFF.

**Deno wasn't installed.** Had to download `deno-x86_64-pc-windows-msvc.zip` into `~/.deno/bin/` to run the unit tests. Supabase's CLI bundles its own Deno for the runtime but doesn't expose it for local scripting. Fine for one-off, but if Deno tests become part of CI, we need an install step.

**Kong overrides CORS in local dev.** Our handler sets `Access-Control-Allow-Origin` only when request Origin matches `APP_BASE_URL`. In local `supabase functions serve`, Kong's reverse proxy rewrites the response `Access-Control-Allow-Origin: *` before it reaches the client, which defeats the strict check. The **code** is correct for hosted/prod; I couldn't verify prod behavior without deploying. Worth a follow-up check once we deploy to the actual Supabase runtime (does it keep our headers, or Kong-style rewrite?). If rewritten, we'd need to configure the Supabase dashboard's CORS allow-list, not just rely on function-layer logic.

**New publishable/secret key format triggers "Legacy token type detected".** Supabase CLI now issues `sb_publishable_...` / `sb_secret_...` keys by default, but `supabase functions serve` logs `Legacy token type detected, attempting HS256 verification` on each request. Verification succeeds — just a noisy warning. Nothing to fix on our side; the CLI is in the middle of a key-format migration.

**Subagent review caught the three should-fixes above** (numeric bounds, RPC narrowing, XFF doc note) plus a handful of justified nits. Review doc lives only in the agent transcript; key findings are reflected above and in the code.

## Decisions

- **Structural validation at edge, semantic at RPC** — avoid drift, single source of truth for email regex / year ceiling / facility existence. Edge codes match RPC codes where they overlap so clients see one code space.
- **`EMAIL_MODE=dev-console` explicit opt-in** over silent fallback — unset `RESEND_API_KEY` in prod fails loudly instead of quietly logging tokens.
- **Log email failure, return `{ok: true}`** — the submission is saved and the rate-limit slot is already consumed; returning error would create a worse UX than silent-success with token expiry. Documented in-code because it's non-obvious.
- **Bounds at the edge match the DB column** (`numeric(10,2)` → 99999999.99 cap) so price overflow returns a proper `invalid_price` instead of `internal_error`. Year bounds intentionally looser than RPC so the edge doesn't need to know `now()`.
- **IP spoofing documented, not fixed** — the per-IP limit is advisory-only on non-Supabase deployments; we don't self-host today. Code comment flags the constraint for future-us.
- **No handler-level integration tests** — unit tests cover the pieces (validation, email, URL construction, Resend shape). Full E2E was run manually via `curl` against `supabase functions serve` + PostgREST. CLAUDE.md allows this for MVP; if we add CI, handler tests become the gap.
